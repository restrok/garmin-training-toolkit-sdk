# Garmin Connect API Quirks & Implementation Notes

This document captures the "ground truth" for interacting with the Garmin Connect internal API, specifically for workout uploads. These findings are derived from reverse-engineering existing workouts and empirical testing.

## Workout Uploads (HTTP 500 / InvalidTypeIdException)

The Garmin API is extremely sensitive to the specific DTO (Data Transfer Object) types and field names used in the workout JSON.

### 1. Repeat Groups (Intervals)
**Issue:** Using `RepeatStepDTO` with `repeatChildSteps` often triggers an `InvalidTypeIdException`.

**Working Implementation:**
*   **Type:** Use `RepeatGroupDTO` instead of `RepeatStepDTO`.
*   **Steps List:** Use `workoutSteps` instead of `repeatChildSteps`.
*   **Condition ID:** Use `conditionTypeId: 7` (iterations) for the repeat block itself.
*   **Structure Example:**
    ```json
    {
      "type": "RepeatGroupDTO",
      "stepOrder": 2,
      "stepType": {"stepTypeId": 6, "stepTypeKey": "repeat", "displayOrder": 6},
      "numberOfIterations": 8,
      "workoutSteps": [...],
      "smartRepeat": false,
      "endCondition": {
        "conditionTypeId": 7,
        "conditionTypeKey": "iterations",
        "displayOrder": 7,
        "displayable": true
      },
      "endConditionValue": 8.0
    }
    ```

### 2. Distance Durations
**Working Implementation:**
*   **Condition ID:** Use `conditionTypeId: 3` for distance-based steps.
*   **Key:** `conditionTypeKey: "distance"`.
*   **Value:** `endConditionValue` should be in meters.

### 3. Target Mapping (Triple Redundancy)
Garmin often requires target information to be duplicated across multiple fields to appear correctly on all devices.
*   **Target Type IDs:**
    *   `1`: `no.target`
    *   `2`: `power.zone`
    *   `4`: `heart.rate.zone`
    *   `5`: `speed.zone` (Pace)
*   **Redundancy Fields:** Always populate `targetValueOne`, `targetValueTwo`, AND the `zone` object (with `low`/`high` keys).

## SDK Implementation Strategy

To minimize LLM errors and token usage, the SDK implements a "Smart Translator":
1.  **LLM Intent:** The LLM provides high-level fields like `distance_m` or `min_bpm`.
2.  **Mapping:** The `create_workout` function in `uploaders/workouts.py` handles the mapping to the quirky Garmin DTOs listed above.
3.  **Validation:** Use the `WorkoutPlan` protocol to validate structures before they are sent to the uploader.

## Maintenance Note
If you encounter a new `InvalidTypeIdException`, use the `debug_activity.py` script to download a known-working workout from the user's account and compare its JSON structure to the one generated by the SDK.
